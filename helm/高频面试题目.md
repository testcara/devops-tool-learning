Helm 高频高级面试题 + 标准答案
1. Helm 3 和 Helm 2 核心区别？（必问）
标准答案

    Helm2 有集群端组件 Tiller，权限很大、安全风险高；Helm3 彻底移除 Tiller，纯客户端运行。
    Helm2 依赖集群存储存 Release；Helm3 用 Secret 存储 Release 记录，更安全。
    Helm3 默认必须指定命名空间，权限模型更严谨。
    废弃 helm init、helm repo add 逻辑优化，OCI 镜像仓库原生支持。
    严格 OCI 标准、runc/containerd 生态对齐更好。

2. Helm 的 Release 是什么？和 Chart 有什么区别？
标准答案

    Chart：静态模板包（代码 / 骨架，存放本地或仓库）
    Release：集群里一次具体的安装 / 升级实例
    每次 install /upgrade 生成一个 Release 版本（revision）
    Helm 通过 Release 保存：渲染后 YAML、values、变更快照，
    以此实现版本追溯、一键回滚、应用生命周期管理。

3. 说下 Helm 模板渲染原理？为什么能多环境？
标准答案

    核心：Go template 引擎 + 内置对象
    渲染链路：
    Chart模板({{ 变量 }}) + Values配置 + 内置对象(.Release/.Chart)
    → 实时渲染成纯净 K8s YAML
    多环境本质：
    同一套 Chart 不变，切换不同 values 文件，实现开发 / 测试 / 生产隔离，避免 YAML 重复编写。

4. 集群出现配置漂移，Helm 怎么解决？
标准答案

    人为手动改集群 YAML 会产生配置漂移；
    Helm 声明式管理：一切变更必须走 upgrade；
    配合 ArgoCD/Flux 做 GitOps，Git 唯一可信源；
    禁止直接 kubectl edit 改资源，从流程上杜绝漂移。

5. helm install 和 helm upgrade --install 区别？生产用哪个？
标准答案

    helm install：只能全新安装，重复执行报错
    helm upgrade --install：幂等操作
    不存在则安装、已存在则升级
    ✅ 生产统一用：upgrade --install

6. Helm 中 subchart / charts 目录 作用？依赖怎么管理？
标准答案

    charts/ 存放子 Chart（依赖组件），比如自己应用依赖 mysql、redis。
    两种依赖管理：
        本地：放入 charts 目录
        远程：Chart.yaml 配置 dependencies，helm dependency update 拉取
    实现整套应用 + 中间件一体化打包部署。

7. values 有几层优先级？（高级高频）
标准答案 优先级从低 → 高

    chart 内置默认 values.yaml
    子 chart values
    自定义 -f 外部values文件
    --set 命令行临时覆盖
    👉 命令行 set 优先级最高，适合临时紧急改参数。

8. Helm 回滚原理？回滚会发生什么？
标准答案

    回滚 = 读取历史 Release 快照（旧 values + 旧模板）
    重新渲染 YAML，向 K8s 提交变更
    控制器滚动重建 Pod，还原配置、镜像、副本数
    适合故障快速止损，是 K8s 原生不具备的能力。

9. Kustomize 和 Helm 区别？面试必问
标准答案

    Helm：
    包管理器 + 模板化 + 版本管理 + 依赖管理，适合完整应用交付、多环境、复杂组件。
    Kustomize：
    原生 K8s 无模板、纯补丁叠加，轻量、无学习成本，适合简单场景、少量差异化。

    总结：复杂云原生应用选 Helm，简单轻量场景选 Kustomize。

10. Helm 有什么缺点？生产痛点？
标准答案

    模板语法（Go template）写复杂逻辑晦涩、难调试。
    大规模多集群场景，纯 Helm 不好治理，必须搭配 GitOps（ArgoCD）。
    不严格约束手动改集群，流程不规范依然会漂移。
    自定义 CRD 管理较弱，需要额外处理 CRD 生命周期。

11. Helm 如何调试模板？你用哪些命令？
标准答案（实操向，你都会）
bash
运行

# 干渲染，不部署，看最终yaml
helm template .

# 查看集群内真实release渲染内容
helm get manifest my-app

# 查看当前生效values
helm get values my-app

12. 简述 Helm + ArgoCD 企业落地架构
标准答案（贴合你 JD）

    用 Helm Chart 封装应用、values 区分多环境；
    所有 Chart & values 统一提交 Git；
    ArgoCD 监听 Git 仓库，自动拉取、渲染 Helm 模板；
    集群状态与 Git 目标状态持续比对，自动修复配置漂移；
    实现声明式交付、可追溯、自动化、多集群统一管理。

极简背诵万能总结（面试收尾必说）
Helm 是 K8s 生态标准应用包管理器，通过 Chart 统一封装所有资源模板，依靠 Values 做多环境配置分离，基于 Release 机制实现应用版本管理、升级与回滚，屏蔽原生 YAML 碎片化问题，结合 GitOps 可实现企业级标准化持续交付。
